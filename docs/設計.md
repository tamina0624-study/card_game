# 設計ドキュメント: AIキャラクターカードバトルゲーム

本ドキュメントは `開発指示書.md`(確定事項・技術スタックを含む)と `システムプロンプト.md` を前提とした、MVP実装のための設計をまとめたものである。

参照:
- 仕様全体: `../開発指示書.md`
- AI審判用システムプロンプト(戦闘生成時にそのまま使用): `../システムプロンプト.md`

技術スタック(確定): Next.js (TypeScript, App Router) + MySQL(PHPブリッジ経由) + Claude API (Anthropic)

> **2026-07-06 追記(DB移行)**: 当初はSQLite(ローカルファイルDB、`better-sqlite3`)で
> 実装していたが、Render等のNext.js稼働先ではディスクが一時的でDB・アップロード画像が
> 永続化できない問題があったため、DBをMySQLに、静的アセット(背景・サンプル画像・BGM・
> アップロード画像)をスターサーバー(`ss181301.stars.ne.jp`)へ移行した。
> スターサーバー等の共有ホスティングは外部からのMySQL直接接続を許可しないことが多いため、
> `public_html/card_game/php` に配置するPHPスクリプト(リポジトリ内 `php/`)がMySQLへの
> アクセスを仲介するAPIブリッジとして働き、Next.js側は `src/lib/bridge/client.ts` 経由で
> HTTPでこれを呼び出す(直接のSQL実行はしない)。以降の1〜4章はSQLite時代の設計を残すが、
> DBスキーマ(2章)は `php/schema.sql`(MySQL DDL)が正、APIの実処理は `php/*.php` が正となる。
> 静的アセットは `src/lib/assets.ts` の `ASSET_BASE_URL`(`NEXT_PUBLIC_ASSET_BASE_URL`)を
> 起点にした絶対URLで配信する。

## 0. MVPスコープに関する前提・追加決定事項

開発指示書の「確定事項」に加え、設計上必要になった以下の点を追加で決定する。実装担当エージェントはこの前提に従うこと。

1. **ユーザー認証は実装しない。** ログイン機能は設けず、キャラクター・デッキは共有プールとして誰でも作成・閲覧・対戦に利用できる(デッキに任意の `ownerName`(自由入力の作成者名)を持たせるのみ)。
2. **前衛/控えの割り当てはデッキ編集画面で行う。** 開発指示書は「戦闘開始時に前衛4枚を選択する」とあるが、実装簡略化のため「デッキ編成時に8枚を登録し、そのうち4枚を前衛・4枚を控えとして指定して保存する」設計とする(`deck_cards.role`)。対戦実行時はデッキに保存された前衛4体で開始する。
3. **画像アップロードは実装時に設計する(指示書 確定事項7)。** 保存先はローカルディスク `public/uploads/characters/`、許容形式は `png/jpg/jpeg/webp/gif`、最大サイズ 2MB、モデレーションは行わない(将来課題として明記)。
4. **戦闘実行は同期API呼び出しとする。** `POST /api/battles` の中で同期的にClaude APIを呼び出し、レスポンスが得られてから結果を返す(MVPでは非同期ジョブ化・SSE配信は行わない)。フロントエンドはローディング状態を表示する。
5. **AI出力のJSONスキーマは開発指示書の「AI出力形式」をそのまま踏襲する。** `events[].type` や `effectType` 等の値は固定enumにせず自由記述を許容し、DB側は生JSON (`raw_json`) も保持して表示側の自由度を確保する。

## 1. アーキテクチャ概要

### 1.1 全体構成

```
Browser (Next.js App Router / React Server Components + Client Components)
        │  fetch (JSON / multipart)
        ▼
Next.js Route Handlers (src/app/api/**)  ── Node.js runtime (better-sqlite3利用のため edge runtime不可)
        │
        ├─ lib/characters, lib/decks, lib/battles ... ドメインロジック層 (バリデーション + リポジトリ)
        │
        ├─ lib/db (better-sqlite3 シングルトン接続) ── data/game.db (SQLiteファイル)
        │
        └─ lib/claude (Anthropic SDK クライアント) ── Anthropic Messages API
```

- フルスタックをNext.js単体(App Router)で構成し、フロントエンドとAPI(Route Handlers)を同一プロジェクトに同居させる。
- データ永続化は SQLite ファイル(`data/game.db`)。ORMは使わず `better-sqlite3` を直接使用する(同期API・軽量・Next.jsのNode.jsランタイムと相性が良い)。
- SQLiteアクセスを行うAPI Routeには `export const runtime = "nodejs"`(App Routerのデフォルト)を明示し、Edge Runtimeでは動かさない。
- Claude API呼び出しは `@anthropic-ai/sdk` の公式SDKを使用し、生の `fetch` は使わない。

### 1.4 Claude API呼び出し方針

- SDK: `@anthropic-ai/sdk`(公式SDK)を使用し、生の `fetch`/`curl` は使わない。
- モデル: 環境変数 `ANTHROPIC_MODEL` で指定可能とし、デフォルトは `claude-opus-4-8`。
- システムプロンプト: `システムプロンプト.md` の内容をそのまま `system` パラメータに渡す(ハードコードせず、サーバー起動時にファイルから読み込みメモリにキャッシュする。ファイル内容を書き換えたら再起動で反映される)。
- ユーザーメッセージ: `lib/battles/prompt.ts` が以下を含めて構築する。
  - チームA/チームBそれぞれの前衛4体・控え4体の「名前・説明・パラメータ名と値・必殺技(技名/説明/演出テキスト)」
  - 戦闘フェーズ(戦闘前分析→戦闘開始→通常行動→必殺技発動→戦闘不能判定→控え補充→勝敗決定→MVP発表)の遂行を指示
  - 確定事項(行動順は完全ランダム、100ポイント制約はアプリ側チェック済みなのでAI側は検証不要、必殺技の発動有無/選択はランダム、控え補充もランダム)を明記
  - 開発指示書「AI出力形式」のJSONスキーマ(analysis/battleLog/events/result)をそのまま提示し、「出力は指定のJSON形式のみとし、Markdownのコードフェンスや説明文を含めないこと」を明示的に指示
- 呼び出しパラメータ: `max_tokens` は 8000 程度(非ストリーミングで十分間に合う範囲)。サンプリングパラメータ(`temperature`/`top_p`/`top_k`)・`thinking` は指定しない(省略可、最新モデルでは省略時の既定動作に委ねる)。ストリーミングは必須ではないためMVPでは非ストリーミング呼び出し(`client.messages.create`)を用いる。
- レスポンス解析:
  1. `response.content` の `text` タイプブロックを結合。
  2. 先頭/末尾の ```` ```json ```` 等のコードフェンスを除去。
  3. `JSON.parse` を試行。
  4. zodスキーマ(`lib/battles/responseSchema.ts`)で構造検証(`events[]` の要素は `type` 以外のキーを自由に許容する `passthrough` 設計)。
  5. 失敗した場合、直前のアシスタント発話+「JSONとして不正だったため、指定のJSON形式のみで出力し直してください」という訂正メッセージを追加して1回だけ再試行する。
  6. 2回失敗した場合は `battles.status = 'failed'` として `error_message` に理由を保存し、APIは502相当のエラーを返す。
- 監査用に、最終的に採用したAI応答の生テキスト(またはJSON文字列)を `battles.raw_ai_response` に保存する。
- `events[]` と `battleLog[]` は仕様上ターン番号で厳密に紐付く保証がないため、`battle_events.turn` はAIの出力に数値の `turn` 相当フィールドがあればそれを採用し、無ければ `NULL` とする(配列順序は `sort_order` で保持)。

## 2. データベーススキーマ (SQLite)

`src/lib/db/schema.sql` としてそのまま実行できるDDL。すべて `CREATE TABLE IF NOT EXISTS` で冪等に実行できるようにする。

```sql
PRAGMA foreign_keys = ON;

-- キャラクター
CREATE TABLE IF NOT EXISTS characters (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  image_url TEXT,
  description TEXT,
  total_points INTEGER NOT NULL DEFAULT 0,     -- パラメータ合計値のキャッシュ(登録/更新時に再計算)
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- キャラクターのパラメータ(名前・値ともに自由入力のため正規化テーブルとする)
CREATE TABLE IF NOT EXISTS character_parameters (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  character_id INTEGER NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  value INTEGER NOT NULL CHECK (value >= 0 AND value <= 100),
  sort_order INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_character_parameters_character_id
  ON character_parameters(character_id);

-- 必殺技(1キャラクターにつき複数保持可)
CREATE TABLE IF NOT EXISTS special_moves (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  character_id INTEGER NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
  name TEXT NOT NULL,           -- 技名
  description TEXT,             -- 説明
  flavor_text TEXT,              -- 演出テキスト
  sort_order INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_special_moves_character_id
  ON special_moves(character_id);

-- デッキ
CREATE TABLE IF NOT EXISTS decks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  owner_name TEXT,               -- 自由入力の作成者名(認証なしのため任意)
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- デッキとキャラクターの紐付け(前衛/控え区分を保持)
CREATE TABLE IF NOT EXISTS deck_cards (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  deck_id INTEGER NOT NULL REFERENCES decks(id) ON DELETE CASCADE,
  character_id INTEGER NOT NULL REFERENCES characters(id),
  role TEXT NOT NULL CHECK (role IN ('front', 'bench')),
  slot_order INTEGER NOT NULL DEFAULT 0,   -- 表示順(前衛内/控え内の並び)
  UNIQUE(deck_id, character_id)
);
CREATE INDEX IF NOT EXISTS idx_deck_cards_deck_id ON deck_cards(deck_id);

-- 対戦
CREATE TABLE IF NOT EXISTS battles (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  deck_a_id INTEGER NOT NULL REFERENCES decks(id),
  deck_b_id INTEGER NOT NULL REFERENCES decks(id),
  status TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'running', 'completed', 'failed')),
  winner TEXT CHECK (winner IN ('teamA', 'teamB')),
  mvp_character_id INTEGER REFERENCES characters(id),  -- ベストエフォートで名前一致解決したID(見つからなければNULL)
  mvp_name TEXT,                                        -- AIが返した生のMVP名(自由記述)
  analysis_team_a TEXT,
  analysis_team_b TEXT,
  predicted_winner TEXT CHECK (predicted_winner IN ('teamA', 'teamB')),
  raw_ai_response TEXT,          -- 監査用: 採用したAI応答の生テキスト
  error_message TEXT,            -- status='failed'の場合の理由
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  completed_at TEXT
);

-- 戦闘ログ(実況テキスト、ターン単位)
CREATE TABLE IF NOT EXISTS battle_logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  battle_id INTEGER NOT NULL REFERENCES battles(id) ON DELETE CASCADE,
  turn INTEGER NOT NULL,
  message TEXT NOT NULL,
  sort_order INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_battle_logs_battle_id ON battle_logs(battle_id);

-- 必殺技演出などのイベント(自由記述フィールドを保持するため raw_json を必須で保存)
CREATE TABLE IF NOT EXISTS battle_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  battle_id INTEGER NOT NULL REFERENCES battles(id) ON DELETE CASCADE,
  turn INTEGER,                  -- AI出力に対応turnが無ければNULL
  event_type TEXT,                -- events[].type (自由記述)
  character_name TEXT,            -- events[].character (自由記述)
  effect TEXT,                    -- events[].effect (自由記述)
  raw_json TEXT NOT NULL,         -- イベントオブジェクト全体(effectType/camera/message等を保持)
  sort_order INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_battle_events_battle_id ON battle_events(battle_id);
```

補足:
- 100ポイント制約はDBのCHECK制約ではなく、アプリ側(`lib/characters/validation.ts`)でキャラクター登録・更新時のみ検証する(確定事項2)。DB上は `character_parameters.value` の 0〜100 の範囲チェックのみ行う。
- デッキの「8枚ちょうど・前衛4/控え4ちょうど」もアプリ側(`lib/decks/validation.ts`)で検証する。

## 3. 主要APIルート一覧

ベースパス: `/api`。すべてJSONレスポンス(画像アップロードのみmultipart/form-dataリクエスト)。

| Method | Path | 概要 | 主なリクエスト | 主なレスポンス |
|---|---|---|---|---|
| POST | `/api/characters` | キャラクター作成 | `{ name, description?, imageUrl?, parameters: [{name, value}], specialMoves?: [{name, description?, flavorText?}] }` | `201 Character`(合計100ポイント超過時は`400`) |
| GET | `/api/characters` | キャラクター一覧取得 | クエリなし | `200 Character[]`(パラメータ合計・必殺技数を含む簡易情報) |
| GET | `/api/characters/:id` | キャラクター詳細取得 | - | `200 Character`(parameters/specialMoves全件含む) |
| PUT | `/api/characters/:id` | キャラクター更新 | 作成と同様のボディ | `200 Character`(100ポイント超過時は`400`) |
| DELETE | `/api/characters/:id` | キャラクター削除 | - | `204`(いずれかのデッキで使用中の場合は`409`) |
| POST | `/api/upload/image` | キャラクター画像アップロード | `multipart/form-data`(`file`) | `200 { url: string }`(2MB超過/非対応形式は`400`) |
| POST | `/api/decks` | デッキ作成 | `{ name, ownerName?, cards: [{characterId, role: 'front'|'bench'}] }`(8件ちょうど、front4/bench4) | `201 Deck` |
| GET | `/api/decks` | デッキ一覧取得 | - | `200 Deck[]`(名前・カード枚数などの概要) |
| GET | `/api/decks/:id` | デッキ詳細取得 | - | `200 Deck`(front/bench各4体のキャラクター全情報を含む) |
| PUT | `/api/decks/:id` | デッキ更新 | 作成と同様のボディ | `200 Deck` |
| DELETE | `/api/decks/:id` | デッキ削除 | - | `204` |
| POST | `/api/battles` | 対戦実行(同期) | `{ deckAId, deckBId }` | `201 BattleDetail`(Claude呼び出し失敗時は`502`、詳細は`errorMessage`) |
| GET | `/api/battles` | 対戦履歴一覧取得 | - | `200 BattleSummary[]` |
| GET | `/api/battles/:id` | 対戦詳細取得 | - | `200 BattleDetail`(analysis/battleLog/events/result一式) |

`BattleDetail` の形状イメージ:

```ts
type BattleDetail = {
  id: number;
  status: 'pending' | 'running' | 'completed' | 'failed';
  deckA: { id: number; name: string };
  deckB: { id: number; name: string };
  analysis: { teamA: string; teamB: string; predictedWinner: 'teamA' | 'teamB' } | null;
  battleLog: { turn: number; message: string }[];
```
