■仕様レビュー(開発指示書.md)
良い点

「自由度」というコンセプトが一貫しており、パラメータ名/必殺技/デッキ構成すべてがそれを支えている
AI判定ルールとJSON出力形式が具体例付きで明確
システムプロンプトと開発指示書の内容(公平性、能力値重視、ストーリー補正禁止など)が一致している
曖昧・未定義で実装に詰まりそうな点

■問題点1
前衛4体の戦闘順序が未定義 — 同時攻撃?ターン制で1体ずつ?速度パラメータで決まる?自由なパラメータ名を許容する以上、「素早さ」的な概念が無いキャラの行動順をどう決めるかのルールが必要。
⇒素早さの概念は作りません。完全にランダムで順番に行動させてください

■問題点2
100ポイント制約をAIがどう検証するか未定義 — ユーザーが自己申告で100を超えて配分した場合、フロントで弾くのかAI審判が無視するのか。
⇒AIで判断しなくていいです。デッキ・カードを登録する時にだけ100ポイント制限をチェックしてください

■問題点3
必殺技の発動条件が未定義 — 毎ターン打てるのか、クールダウンやゲージがあるのか。「複数の必殺技を持てる」だけで選択ロジックがない。
⇒細かい設定は付けないので、ランダムで発生させてください。選択も必殺技の発生時にランダムでそのキャラクターの必殺技を選択してください。

■問題点4
控え補充のタイミングと選出ルールが未定義 — 前衛の誰かが戦闘不能になった瞬間に控えのどれが出るか(プレイヤー選択?自動?)。
⇒自動です。残っているキャラクターからランダムです。

■問題点5
JSON出力のスキーマが厳密でない — winnerが"teamA"固定なのか、実際のチーム名/IDを使うのか。events[].typeやeffectTypeの許容値一覧(enum)が未定義で、フロントエンドの演出実装がAIの自由記述に依存してしまう。
⇒AIの自由記述に依存して大丈夫です。

■問題点6
AIの決定性・再現性の要件がない — 同じ2チームで対戦させたとき、多少の結果ブレを許容するのか、シード固定で再現可能にするのか(バグ検証やランキング公平性に影響)。
⇒結果のブレは許容します。再現性がなくてもいいです。

■問題点7
画像アップロードの扱いが未定義 — 保存先、モデレーション、サイズ制限など。
⇒実装時に設計してください。お任せします

■問題点8
対戦相手のマッチング/バランス調整のルールがない — 100ポイント固定なので理論上バランスは取れるはずだが、「パラメータ名の説得力」次第でAIの解釈が有利不利を生む(例:「世界最強100」のような曖昧ワードをどう抑制するか、システムプロンプト側にも具体的なガイドが少ない)。
⇒厳密なルールはいりません。AIの解釈に任せていいです。

開発体制(AI駆動設定.md)について
「夜間に複数エージェントで自律継続」「30分粒度分割」という指示は、実際にオーケストレーションする場合は Workflow ツールでの実装が必要になります(現状では明示的な「ワークフローを使って」という依頼がない限り私からは起動しません)。またこれは実行環境依存(使用量制限で停止、進捗ファイル更新)なので、CLAUDE.md的な運用ルールとして明文化しておくと良さそうです。
⇒ワークフローを使ってください。
